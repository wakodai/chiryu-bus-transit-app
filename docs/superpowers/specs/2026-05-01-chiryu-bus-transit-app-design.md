# 知立市ミニバス 乗換検索Webアプリ 設計

## 1. 目的とスコープ

愛知県知立市のコミュニティバス「ミニバス」全6路線を対象とした、地図ベースの乗換検索Webアプリを構築する。

**ユーザー操作**：地図上で出発地と到着地をクリック → 自動的に最寄りバス停を選定 → 出発時刻指定で経路を検索 → 結果を地図上にハイライト表示。

**実装フェーズ**：当面はローカルサーバー（`npm run dev`）で動作確認できれば良い。将来的にGitHub Pages 等の静的ホスティングに公開する想定。

## 2. データソース

| 項目 | 内容 |
| --- | --- |
| 提供元 | 知立市（GTFSデータリポジトリ経由で配信） |
| 形式 | GTFS-JP 第3版 |
| 直リンク | `https://api.gtfs-data.jp/v2/organizations/chiryucity/feeds/communitybus/files/feed.zip?rid=current` |
| ライセンス | CC BY 2.1 JP |
| 有効期間 | 2025-12-23 〜 2026-12-31 |
| 規模 | 6 routes / 106 stops / 110 trips / 3,109 stop_times（圧縮zip 約83KB） |

**出典表示**：画面下部に「知立市ミニバス GTFS-JPデータ／知立市 (CC BY 2.1 JP)」を常時表示する。

## 3. アーキテクチャ

完全クライアントサイドのSPA。バックエンドは持たない。GTFS zipはビルド時にスクリプトでJSONに変換して `public/gtfs/` に配置し、ブラウザは初回起動時にfetchして全データをメモリ展開する。

```
[Vite + TypeScript + Leaflet SPA]
        │
        ├─ public/gtfs/*.json
        │     stops.json / trips.json / stop_times.json
        │     calendar.json / calendar_dates.json
        │     transfers.json / routes.json / shapes.json
        │     feed_info.json
        │
        └─ scripts/fetch-gtfs.ts
              GTFS zipダウンロード → CSV解析 → JSON出力
              （npm run build:gtfs で再生成）
```

**この構成を選ぶ理由**：データが極めて小さい（数百KB）ので全てクライアントで処理可能。サーバー不要のため運用コストゼロ、デプロイ先の選択肢も広い。

## 4. モジュール構成（src/）

| モジュール | 責務 |
| --- | --- |
| `data/loader.ts` | JSON群をfetchして型付きオブジェクトに展開する |
| `data/index.ts` | バス停の空間インデックス（rbush）と `stop_id × service_id × time` インデックスを構築 |
| `routing/calendar.ts` | 指定日の有効 `service_id` 集合を `calendar.txt` ＋ `calendar_dates.txt` から解決 |
| `routing/nearest.ts` | 任意の緯度経度から半径500m以内のバス停を距離順で返す（最大3件） |
| `routing/raptor.ts` | RAPTOR風アルゴリズムで最大2乗換までの経路を探索し、Paretoフロント（到着時刻×乗換数）を返す |
| `routing/select.ts` | Paretoフロントから「最早着」「最少乗換」「直行（あれば）」の最大3案を選定 |
| `map/view.ts` | Leaflet 初期化、OSMタイル、出発/到着ピン、選択経路の shape ポリラインと番号付きバス停マーカー |
| `ui/app.ts` | 地図クリック受付、日時ピッカー、検索ボタン、結果リスト表示の総合制御 |
| `ui/result.ts` | 検索結果（時刻表・乗換・所要時間）のレンダリングと地図ハイライト連動 |

各モジュールは型付きインターフェースで分離し、単独でテスト可能とする。

## 5. データフロー（ユーザー操作）

1. 地図クリック1回目 → 出発地ピン設置、半径500m以内の最寄りバス停をハイライト
2. 地図クリック2回目 → 到着地ピン設置、最寄りバス停をハイライト
3. 出発時刻ピッカー（デフォルト=現在時刻）と日付ピッカー（デフォルト=今日）
4. 「検索」ボタン押下 → RAPTOR検索 → Paretoフロント抽出 → 最大3案を結果パネルに表示
5. 結果カードを選択 → 地図上で該当経路の shape をポリライン強調、各停車バス停に乗車順番号を表示

ピンは「リセット」「入れ替え（出発⇄到着）」操作可能。

## 6. 乗換検索アルゴリズム

### RAPTOR の動作

ラウンド k = 「最大 k 回の乗継で各バス停に到達可能な最早到着時刻」を更新する。

- ラウンド 0：出発地から徒歩到達可能な各バス停の「出発可能時刻」を初期化
- ラウンド 1：上記バス停を出発する全 trip を辿り、後続バス停の到着時刻を更新
- ラウンド 2：ラウンド1で到達したバス停＋徒歩接続から、もう1本のバスで到達可能なバス停を更新
- ラウンド 3：もう1乗換（最大2乗換まで）

各バス停で「(到着時刻, 乗換数)」のPareto最適解集合を保持する（単純な最早到着のみだと、より少ない乗換数の解が消える）。

### 徒歩接続

- 出発地→最寄りバス停、最寄りバス停→到着地：直線距離 100m あたり 1 分（80m/分の標準歩行速度に近似）
- バス停間の徒歩乗継：`transfers.txt` に明示された接続のみを使用（`min_transfer_time` を遵守）

### 結果選定

Paretoフロントから3案を抽出する：
1. 最早到着案
2. 最早到着案の中で乗換数が最小の案（重複時はスキップ）
3. 直行案（あれば、かつ上記2案と重複しないもの）

3案未満になる場合はそのまま返す。0件なら「経路が見つかりません」を表示。

### 仕様の制約

- 同一日内の検索のみ（日付跨ぎ・翌日始発検索は対象外）
- 最大乗換回数：2回（=ラウンド3まで）
- 検索範囲：指定日に運行する trip のうち、指定出発時刻以降に出発するもの

## 7. エラー/エッジケース

| ケース | 挙動 |
| --- | --- |
| 半径500m以内にバス停なし | 「近くにバス停がありません」を表示し、半径を1kmに広げる選択肢を提示 |
| 経路なし | 「経路が見つかりません。日時/地点を変えて再検索してください」 |
| 始発前/終発後 | 「本日の運行は終了しています」または「次の始発は◯時◯分です」と案内 |
| GTFSデータfetch失敗 | ビルド時に同梱したフォールバックJSONを使用し、画面上部に通知バナーを表示 |
| 運休日（年始等） | calendar_dates.txt の `exception_type=2` を尊重して該当serviceを除外 |

## 8. 公開とローカル実行

- **ローカル**：`npm run dev` → `http://localhost:5173`
- **データ更新**：`npm run build:gtfs`（zipを再ダウンロードしてJSONに再変換）
- **ビルド**：`npm run build` → `dist/` を任意の静的ホスティングへデプロイ可能
- **公開先候補**：GitHub Pages / Cloudflare Pages / Netlify

## 9. テスト方針

| 種類 | 対象 |
| --- | --- |
| ユニット | `routing/calendar.ts`（特定日付に対する有効service_id解決） |
| ユニット | `routing/nearest.ts`（既知座標から想定バス停が返ること） |
| ユニット | `routing/raptor.ts`（fixture経路で「直行」「乗継」両方の正解が取れること） |
| ユニット | `routing/select.ts`（Pareto集合からの3案選定） |
| 結合 | dev server を起動し、ブラウザで実際の検索シナリオを手動確認 |

テストランナー：Vitest（Viteと統合）。

## 10. YAGNIで除外する機能

- 運賃計算 — 知立市ミニバスは一律100円のため「100円 × 乗車回数」を結果に併記するだけ
- リアルタイム位置情報 — GTFS-RT未提供のため
- ユーザーアカウント、検索履歴
- スマホ専用UI — 通常のレスポンシブ対応のみ
- 多言語化 — 日本語のみ
- 翌日始発検索、長期間検索

将来的に必要になったら別specで設計する。
